# 商用化フィージビリティレポート（Weather You Travel）

作成日: 2026-06-10
対象: 本プロジェクトを個人企業として実サービス化する場合の検討資料

---

## 0. エグゼクティブサマリー（結論）

| 観点 | 評価 | 要約 |
|---|---|---|
| 技術的な完成度 | ★☆☆ | プロトタイプとしては良いが、商用化にはバックエンド層の新設が必須（APIキー露出・HTTPS問題が致命的） |
| 需要 | ★★☆ | 「便名→現地天気」単体の需要は弱い。競合（Google検索・Flighty・航空会社アプリ）が強く、差別化はニッチ |
| ビジネスとしての期待値 | ★☆☆ | 単体での収益化は困難。アフィリエイト併用で月数千円〜数万円が現実的な上限 |
| 低コスト自動運用の実現性 | ★★★ | 無料枠＋AIエージェント自動化で月額 約500〜3,500円から検証可能。「ほぼ放置」運用は実現可能 |
| **総合判断** | **条件付きGo** | 事業としてではなく「ほぼゼロコストの実験」として進める価値はある。3ヶ月の検証期間とKPIによるGo/No-Go判断を推奨（§7） |

**推奨方針**: 初期投資を月3,500円以下に抑え、運用をAIエージェントにほぼ全面委任した状態で3ヶ月公開し、SEO流入と利用数で需要を実測する。数値が基準を超えたら有料API・収益化に投資する。

## 1. システムとして足りない部分（技術的ギャップ）

現状のコードベース（React + Vite、`src/services/api.js` にモック＋実API二重化）を精査した結果、商用運用には以下が不足している。

### 1-1. 致命的（公開前に必須）

| 問題 | 詳細 | 対策 |
|---|---|---|
| **APIキーのクライアント露出** | `VITE_AVIATIONSTACK_KEY` 等はビルド時にJSバンドルへ埋め込まれ、誰でも抽出できる。月100リクエストの無料枠は即座に第三者に使い切られる | バックエンド（Cloudflare Workers等）でAPIプロキシを作り、キーをサーバー側に隠蔽する |
| **AviationStackがHTTP接続** | コード内のURLが `http://api.aviationstack.com`。HTTPSで配信するサイトからは混在コンテンツとしてブラウザにブロックされ、**本番では一切動作しない**（無料プランはHTTPSが使えないため、プロキシ経由が必須） | 同上。Workers経由でHTTPS化 |
| **キャッシュ・レート制限なし** | 同じ便名・都市の検索でも毎回APIを呼ぶ。無料枠100req/月では1日3検索で枯渇 | Workers KVで便名・都市単位のキャッシュ（フライトスケジュールは24h、天気は1〜3h）。これだけで実質的な利用可能数が10倍以上になる |

### 1-2. 重要（コア価値に関わる）

| 問題 | 詳細 |
|---|---|
| **滞在期間の天気ロジックが未実装** | アプリの売りは「滞在期間中の現地天気」だが、現状は復路の日付を一切見ておらず、固定3日分を表示するだけ。往路到着日〜復路出発日から日数を算出する処理が必要 |
| **予報期間の制約** | OpenWeatherMapの標準予報は5日先まで。「来月の旅行」には使えない。One Call API 3.0のdaily（8日）か、Open-Meteo（16日・無料）等への切替を検討すべき。**正確性が最重要という要件に対し、出発が6日以上先の旅行をカバーできないのは本質的な欠陥** |
| **都市名の特定が脆弱** | 到着都市をタイムゾーン文字列から推定している（`America/Los_Angeles` → "Los Angeles"）。`Asia/Tokyo` 着の地方空港など、空港所在地とタイムゾーン名が一致しないケースで天気取得が破綻する。IATAコード→都市・緯度経度のマスタデータ（OurAirports等の無料データ）を持つべき |
| **エラーハンドリング/リトライ未実装** | TASKS.mdでも未完了。API障害時は静かにモックへフォールバックするため、**ユーザーに偽のデータを見せるリスク**がある。商用ではモックフォールバックは無効化し、明示的なエラー表示に変えるべき |

### 1-3. 品質・運用面

- **テストが形骸化**: `App.test.jsx` はレンダリング確認のみ、E2Eはサンプルのまま。自動運用（後述）の前提となる「CIが壊れたら検知できる」状態にするため、コア機能（検索→表示、エラー表示）のテストが必要
- **バグ**: `api.js` の `getWeatherIcon` のdefault分岐が `'tj'` という文字列を返している（タイポ）。未知の天気でそのまま画面に出る
- **監視ゼロ**: エラー監視（Sentry無料枠）、死活監視（UptimeRobot無料枠）が未導入。自動運用の生命線なので必須
- **法務系の欠如**: プライバシーポリシー・利用規約・特定商取引法表記（有料化する場合）・問い合わせ窓口がない。広告/アフィリエイトを貼るなら公開前に必要
- **SEO未対応**: SPAでメタデータが静的。流入戦略（§4）の柱がSEOなら、プリレンダリングか静的生成への移行が必要

## 2. ビジネスとして必要な検討

### 2-1. 競合環境

- **Google検索**: 便名を入れるだけで運航状況が出る。天気も都市名検索で即出る。「情報を調べる」需要はGoogleがほぼ吸収済み
- **Flighty / FlightAware / 航空会社アプリ**: リアルタイム追跡・遅延通知では勝負にならない（PROJECT_KNOWLEDGEの方針どおり、リアルタイム性で戦わないのは正しい）
- **天気アプリ各種**: 旅行先の天気を見る行為自体は既存アプリで完結する

### 2-2. 差別化の余地

唯一の差別化は「**便名2つを入れるだけで、滞在期間の現地天気がまとまって出る**」という統合体験。ここから自然に拡張できる価値は:

1. **持ち物・服装の提案**（天気×期間×目的地から生成。AIで安価に実装可能）
2. **旅程へのカレンダー登録**（ics出力は実装コストほぼゼロ）
3. **日本語×インバウンド/アウトバウンド特化**（英語圏ツールが手薄な領域）

逆に言うと、この統合体験＋提案機能まで作らないと、Google検索2回との差が説明できない。

### 2-3. 収益モデルの現実性

| モデル | 現実性 | 備考 |
|---|---|---|
| アフィリエイト（ホテル・eSIM・旅行保険・スーツケース等） | ◎ 最有力 | 目的地と日程が確定しているユーザーは購買意欲が高い。楽天トラベル/Booking.com/Agodaのアフィリエイトと相性が良い |
| 広告（AdSense等） | △ | 単価が低く、月間数万PVないと数千円にもならない |
| プレミアムプラン | ✕（当面） | 無料代替が多すぎて課金理由を作れない。検証フェーズでは考えない |

### 2-4. 法務・運用上の検討事項

- プライバシーポリシー（GA等の計測を入れるなら必須）、利用規約（情報の正確性の免責が特に重要：**天気・運航情報の誤りで損害が出ても責任を負わない旨**）
- アフィリエイト導入時はASP規約・景表法（PR表記）への対応
- 個人でやる場合の事業届・確定申告（収益が出始めてからで可）

## 3. 需要分析 — 進める価値はあるか

### 3-1. 市場シグナル

- 旅行アプリ市場自体は成長中（オンライン旅行予約市場は2030年に約2.5兆ドル規模との予測）で、AIによる旅程パーソナナライズがトレンド
- 一方、フライト追跡アプリ群は広告過多で評価が下がっており、「軽くて1つのことをやるツール」への不満由来の隙間はある
- 「行き先 天気 服装」「○月 ハワイ 服装」系の検索は安定した検索ボリュームがある日本語SEOの定番領域で、ここがこのアプリの現実的な入口

### 3-2. 正直な評価

- **アプリ単体をマーケティングして使ってもらう需要: 弱い**。便名入力という体験は便利だが、年に数回しか旅行しない人にアプリを覚えてもらうのは難しい
- **SEOコンテンツの着地ページとしての需要: 中程度**。「目的地×時期×服装/天気」記事から「あなたの便名で正確にチェック」へ誘導する導線なら、検索流入→ツール利用→アフィリエイトという現実的なファネルが描ける
- 期待収益は楽観でも月数千円〜数万円。**生活費を賄う事業にはならないが、運用コストを月数千円以下に抑えられるなら期待値はプラスにできる**

### 3-3. 結論

「事業」としてはNo-Go寄り、「自動運用の実験＋小さな副収入」としては**Go**。判断を間違えないために、§7のKPIで3ヶ月後に機械的に判定することを推奨する。

## 4. 【第一案】AIエージェント活用による省力運用プラン

「サービス運用・システム改善に労力をかけない」を最優先に、人間の作業を**週30分以内のレビューのみ**に抑える構成。

### 4-1. 推奨アーキテクチャ（運用ゼロ志向）

```
ユーザー → Cloudflare Pages（静的ホスティング・無料）
              ↓
         Cloudflare Workers（APIプロキシ・無料枠 10万req/日）
              ├─ Workers KV キャッシュ（フライト24h / 天気1-3h）
              ├─ AviationStack（キーはWorkersのSecretに保管）
              └─ OpenWeatherMap / Open-Meteo
```

- サーバー管理・OSパッチ・スケーリングが一切不要（サーバーレス）
- キャッシュにより外部API課金を最小化
- この構成自体が「運用労力ゼロ」の最大の施策

### 4-2. AIエージェントによる自動化レイヤー

| 業務 | 自動化方法 | 人間の関与 |
|---|---|---|
| **障害対応** | Sentry（エラー監視・無料枠）→ GitHub Issue自動起票 → Claude Code GitHub Actions（`@claude`メンション or Issueラベルトリガー）が原因調査・修正PR作成 → CIパス確認 | PRレビューとマージ承認のみ |
| **依存パッケージ更新** | Renovate/Dependabot がPR作成 → CIテスト → Claude Codeにレビューさせ、パッチ/マイナーは自動マージ | メジャーバージョンのみ確認 |
| **機能改善** | 要望をGitHub Issueに書く → Claude Codeが実装PR作成 | Issue起票とレビュー |
| **SEO記事生成** | GitHub Actionsの定期実行（cron）でClaude APIを呼び、「目的地×時期の天気・服装」記事を生成して下書きPR | 週1回まとめて確認・公開 |
| **死活監視** | UptimeRobot（無料）→ 障害時にメール通知 | 通知を見るだけ |
| **問い合わせ対応** | 問い合わせフォーム→ メール → Claude APIで返信下書き生成 | 送信前確認 |

ポイント: **自動化の前提は「CIが信頼できること」**。§1-3のテスト整備（コア機能のVitest + Playwright E2E）が自動運用の必須投資になる。テストが無いままエージェントに自動修正させるのは事故のもと。

### 4-3. 段階導入

1. **Phase A（検証期）**: Cloudflare移行 + Workersプロキシ + テスト整備 + Sentry/UptimeRobot + Claude Code GitHub Actions導入
2. **Phase B（流入獲得期)**: SEO記事の定期自動生成、アフィリエイト導入
3. **Phase C（数値が出たら）**: AviationStack有料化、持ち物提案などのAI機能追加

## 5. コスト試算（月額）

### 5-1. 検証フェーズ（〜3ヶ月、無料枠中心）

| 項目 | 月額目安 | 備考 |
|---|---|---|
| ドメイン | 約150円 | 年1,500〜2,000円の按分 |
| Cloudflare Pages / Workers / KV | 0円 | 無料枠（Workers 10万req/日）で十分 |
| AviationStack | 0円 | 無料100req/月＋24hキャッシュで延命。スケジュール検索が主目的なら耐えられる |
| OpenWeatherMap One Call 3.0 | 0円 | 1,000call/日まで無料（要カード登録）。Open-Meteoなら登録すら不要で16日予報 |
| Sentry / UptimeRobot / GitHub | 0円 | すべて無料枠 |
| **AIエージェント（Claude）** | 約3,000円（$20） | Claude Proサブスクリプション（Claude Code利用）。API従量の場合、Opus 4.8は入力$5/出力$25 per 1Mトークンで、週次の自動修正・記事生成程度なら月$5〜20 |
| GitHub Actions | 0円 | パブリックリポジトリなら無料、プライベートでも2,000分/月無料 |
| **合計** | **約500〜3,500円** | Claudeを既に契約済みなら実質ドメイン代のみ |

### 5-2. 成長フェーズ（流入が増えた場合）

| 項目 | 月額目安 | 発動条件 |
|---|---|---|
| AviationStack Basic | 約7,500円（$49.99、10,000req/月） | キャッシュ込みで月100検索を超えたら |
| OpenWeather超過分 | 数百円〜 | 1,000call/日超過分は従量（0.14EUR/100call） |
| Claude Max もしくはAPI増額 | 約7,500〜30,000円 | 自動化ジョブが増えた場合 |
| **合計** | **約15,000〜40,000円** | この段階ではアフィリエイト収益との損益で判断 |

### 5-3. 損益分岐の目安

- 検証フェーズ: 月3,500円 ≒ ホテルアフィリエイト成約1〜2件で回収可能な水準
- 成長フェーズ: 月2〜4万円 ≒ 月間1〜2万UU規模の検索流入が必要（日本語SEOで1年がかりの水準）。**ここに到達する見込みがKPIで見えない限り有料プランへ移行しない**ことが防衛線

## 6. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| AviationStack無料枠の枯渇・精度不足 | コア機能停止 | キャッシュ徹底＋検索回数の上限表示。スケジュール特定だけなら、将来的にOpenSkyや時刻表データ（GTFS的な静的データ）併用も検討 |
| Googleのゼロクリック化でSEO流入が想定以下 | 収益化不能 | 3ヶ月KPIで撤退判断（サンクコストが月数千円なので傷は浅い） |
| AI自動修正PRによるデグレ | サービス品質低下 | テストカバレッジを先に確保。自動マージはパッチ更新のみに限定し、機能変更は必ず人間レビュー |
| 天気・運航情報の誤りによるクレーム | 信頼失墜 | 免責を明記した利用規約、データ取得時刻の表示、「最終確認は公式で」の注意書き |

## 7. 結論と推奨アクション

### Go/No-Go判定基準（公開から3ヶ月後）

| KPI | Go基準（継続・投資） | No-Go基準（凍結） |
|---|---|---|
| 月間UU | 1,000以上 | 100未満 |
| 検索流入（オーガニック） | 月500セッション以上で増加傾向 | 横ばい・減少 |
| 検索実行率（訪問→便名検索） | 20%以上 | 5%未満 |
| 運用工数 | 週30分以内を維持 | 週2時間超が常態化 |

### 直近のアクション（優先順）

1. **Workersプロキシ＋キャッシュ＋HTTPS化**（§1-1。これ無しでは公開不可）
2. **滞在期間ロジックと都市マスタの実装**（§1-2。コア価値そのもの）
3. **コア機能のテスト整備 → CI**（自動運用の前提）
4. **Sentry / UptimeRobot / Claude Code GitHub Actions導入**（運用自動化）
5. **モックフォールバック廃止＋エラーUI、`getWeatherIcon`のタイポ修正**
6. 利用規約・プライバシーポリシー・問い合わせ導線
7. 公開 → SEO記事の自動生成を開始 → 3ヶ月計測
